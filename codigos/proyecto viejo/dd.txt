🧠 ESTRUCTURA RAÍZ ORIGINAL (CONCEPTO FUNDACIONAL)
Cuando pasamos de MIT App Inventor a Flutter, definimos que la app no iba a ser “pantallas sueltas”, sino un sistema jerárquico.
La base es esta:
INSTITUCIÓN
  └── TEMPORADA
        └── TORNEO
              └── CATEGORÍA
                    └── FIXTURE
                          └── PARTIDO
                                └── EVENTOS
________________________________________
🔵 1. NIVEL MÁXIMO — INSTITUCIÓN
Ejemplo:
•	San Fernando
👉 Esto existe porque a futuro:
•	podrías tener múltiples clubes
•	o exportar la app
________________________________________
🟣 2. TEMPORADA
Ejemplo:
•	2026
👉 Importante porque:
•	separa históricos
•	permite comparar años
________________________________________
🟠 3. TORNEO
Ejemplo:
•	Local
•	Nacional (Puede ser Rama A, B, C, etc)
•	Otros
👉 Esto permite:
•	múltiples competencias simultáneas
•	separar datos correctamente
________________________________________
🟡 4. TIPO DE TORNEO
Dentro de cada torneo:
•	Apertura (Especial para Local)
•	Clausura (Especial para Local)
•	Unico (Especial para Nacional A, B, etc)
👉 Esto lo simplificamos como:
torneo = "Apertura" / "Clausura"
________________________________________
🟢 5. CATEGORÍA
Ejemplo:
•	Cadetes
•	Juveniles
•	Otras
👉 Acá definimos algo MUY importante:
•	Cada categoría tiene su propio fixture
•	Pero comparten institución y torneo
________________________________________
🔵 6. FIXTURE
Ejemplo: 
•	Fixture que ya existe
•	Creacion de uno manual, para NACIONAL
•	Creacion/subida manual para x torneo
Es:
👉 la lista de partidos predefinidos
Contiene:
•	fechas
•	rivales
•	local/visitante
•	Torneo
________________________________________
🔴 7. PARTIDO (NÚCLEO)
Esto fue lo más importante del diseño.
Un partido tiene TODO:
•	rival
•	fecha
•	estado
•	goles
•	eventos
•	convocados
•	estadísticas
👉 Es la unidad central del sistema
